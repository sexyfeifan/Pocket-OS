## R01 — 导出完整 TML 文本

### 描述

在选题卡编辑页中，新增「导出 TML」按钮，将当前选题的完整排期信息导出为结构化的纯文本格式。

### 导出内容

导出的 TML 文本须包含以下全部信息：

1. **工序 + 时间**：每行一个工序，格式为 `工序名 日期` 或 `工序名 开始日期~结束日期`
2. **日历备注**：在对应日期行下方以缩进或括号形式附带备注内容
3. **未匹配的备注项**：仅在日历上创建、不属于标准工序的备注，按日期顺序排列

### 输出格式示例

```
大纲 7.12
脚本 7.13
拍摄 7.14~7.17
  (罗山外景，导演：刘帅宏)
粗剪 7.18
A Copy 7.19
调色 7.20
B Copy 7.21
VO 7.22
发布 7.23
7.15 收到品牌寄样
7.25 返图给客户
```

### UI 要求

- 按钮位置：与现有「粘贴 TML」按钮并列
- 点击后弹出文本框，显示导出内容，支持一键复制
- 文本框内可编辑（方便用户调整后再分享）

### 边界情况

- 无备注的工序：仅输出 `工序名 日期`
- 日期范围为单天：输出 `工序名 7.20`，不输出范围格式
- 备注内容含换行：保留换行，每行前加缩进前缀

## R02 — TML 粘贴解析时同步记录原文

### 描述

当前「粘贴 TML → 解析并应用」和「仅记录」是两个独立操作。需求为：「解析并应用」执行时，自动同时将粘贴的原始 TML 文本保存到 `topic.tmlNote` 字段，与「仅记录」的存储方式完全一致。

### 行为定义

| 操作 | 解析工序 | 应用工序 | 记录原文到 tmlNote |
|------|---------|---------|-------------------|
| 解析并应用 | ✅ | ✅ | ✅ **（新增）** |
| 仅记录 | ❌ | ❌ | ✅（现有行为） |

### 实现要点

- `applyTml()` 函数（`index.html:2189`）执行时，在应用工序和备注后，同时将 `tml-input` 的原始文本赋值给 `topic.tmlNote`
- 存储格式与「仅记录」模式完全一致，均为 `topic.tmlNote = "原始文本"`
- 如果已有 `tmlNote`，覆盖还是追加：**覆盖**（以最新一次为准）

## R03 — TML 解析应用时同步更新发布日期

### 描述

当前 `parseTml()` 解析出的工序中如果包含「发布」工序，其日期仅应用到 `productionSteps` 中的发布工序行，但不会同步更新选题卡的 `topic.publishDate` 字段。

### 行为定义

当 TML 解析结果中包含发布工序（`stepKey === 'publish'`）时：

1. 将解析到的发布日期赋值给 `topic.publishDate`
2. 甘特图日历中的发布色块和选题卡右上角的发片日期同步更新
3. 左侧边栏选题列表中的发布日期同步更新

### 实现要点

- 在 `applyTml()` 函数中，应用工序后检查是否存在 `publish` key
- 若存在，执行 `topic.publishDate = tmlParsed.steps.publish.startDate`
- 若解析结果中没有发布工序，保持 `topic.publishDate` 不变

## R04 — 完整操作日志记录

### 描述

为所有数据变更操作建立审计日志，记录操作来源和变更内容。

### 日志字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `timestamp` | ISO 8601 | 操作时间 |
| `ip` | string | 客户端 IP 地址（优先 X-Forwarded-For） |
| `userAgent` | string | 完整 User-Agent 字符串 |
| `fingerprint` | string | 浏览器指纹（前端生成，基于 canvas/WebGL/屏幕分辨率等） |
| `action` | enum | 操作类型（见下表） |
| `topicId` | string | 涉及的选题 ID |
| `detail` | object | 变更详情（diff 或描述） |

### 操作类型枚举

| action 值 | 含义 |
|-----------|------|
| `topic.create` | 新建选题 |
| `topic.delete` | 删除选题 |
| `topic.update` | 修改选题基础信息（标题、分类、平台） |
| `step.toggle` | 工序完成状态切换 |
| `step.update` | 工序日期修改 |
| `step.reorder` | 工序顺序调整 |
| `step.add` | 新增工序 |
| `step.delete` | 删除工序 |
| `date.publish` | 发布日期修改 |
| `prep.add` | 新增前置任务 |
| `prep.toggle` | 前置任务完成切换 |
| `note.create` | 创建日历备注 |
| `note.update` | 修改日历备注 |
| `note.delete` | 删除日历备注 |
| `tml.apply` | TML 解析并应用 |
| `tml.record` | TML 仅记录 |
| `canbox.import` | 从 Canbox 导入 |
| `canbox.bind` | 绑定 Canbox 通告 |
| `settings.save` | 保存工序设置 |
| `data.import` | 导入 JSON 数据 |
| `data.export` | 导出数据 |

### 存储方案

- 日志存储在 `data/activity_log.json`，与 `schedule_data.json` 同级
- 日志文件按日期自动轮转：`activity_log_2026-07-14.json`
- 单文件上限 5MB，超出后自动创建新文件
- 保留最近 30 天的日志文件

### API 设计

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/log` | 前端提交一条日志 |
| GET | `/api/logs?date=2026-07-14&page=1` | 查询指定日期的日志（分页） |

### 前端实现

- 在 `saveData()` 的防抖回调中，收集本次变更的 diff，附带浏览器指纹一起提交
- 浏览器指纹生成：使用轻量方案（canvas fingerprint + 屏幕参数 + UA hash），不引入第三方库
- 日志提交为 fire-and-forget（`navigator.sendBeacon`），不阻塞主流程

### 日志查看入口

- 在侧边栏底部「系统信息」区域新增「操作日志」按钮
- 点击弹出日志列表弹窗，按时间倒序显示
- 支持按操作类型筛选

## R05 — 卡片日历备注双向同步到主日历

### 描述

当前备注仅存储在 `topic.notes[date]` 中，在选题卡内的月历和主页面甘特图日历中分别独立渲染。需求为：在任意位置创建/编辑/删除备注，所有展示该备注的位置同步更新。

### 涉及位置

| 位置 | 当前状态 | 目标状态 |
|------|---------|---------|
| 选题卡内月历（`renderProjectCalendar`） | ✅ 显示备注 | ✅ 保持 |
| 选题卡内编辑页的甘特图（`edit-gantt-container`） | ❌ 不显示备注 | ✅ 显示备注标记 |
| 主页面甘特图日历（`gantt-container`） | ❌ 不显示备注 | ✅ 在对应日期单元格显示备注标记 |

### 实现要点

- 甘特图日期单元格（`date-cell`）渲染时，检查当前选题在该日期是否有 `notes[date]`
- 若有备注，在单元格顶部显示一个小圆点标记（品牌色），hover 显示备注摘要
- 在甘特图日期单元格点击弹出的工序选择弹窗中，底部增加备注区域：
  - 已有备注：显示摘要 + 「编辑」按钮
  - 无备注：显示「添加备注」入口
- 备注的增删改操作执行后，调用 `renderAll()` 刷新所有视图（当前已如此，确保 `renderProjectCalendar` 和 `renderCalendar` 均被调用）

### 甘特图备注标记样式

```
┌──────────┐
│ 7  •      │  ← 小圆点表示有备注
│ [拍摄]    │
└──────────┘
```

- 圆点颜色：`#B37D56`（品牌色）
- 尺寸：6×6px
- 位置：日期数字右侧

## R06 — 工序拖拽排序（按住上下移动）

### 描述

在选题卡编辑页的「制作流程」区域，支持通过拖拽方式调整工序的先后顺序，调整后日历中的工序色块位置自动跟随变动。

### 交互方式

**桌面端：**
- 鼠标按住工序行左侧的拖拽手柄（⠿ 图标），上下拖动
- 拖动过程中目标位置显示插入线
- 释放后工序顺序更新，所有工序的起止日期自动重新计算

**移动端：**
- 长按工序行（500ms），触发拖拽模式
- 上下滑动调整位置
- 释放后同上

### 重排后的日期重算逻辑

工序顺序变更后，以当前 `publishDate` 为锚点，按新顺序重新执行倒排算法 `reverseSchedule(publishDate)`。

**已标记完成的工序：**
- 保留 `done: true` 状态
- 起止日期按新顺序重新分配
- 若已完成工序的新日期落在今天之前，不触发冲突预警（因为已 done）

### UI 要求

- 拖拽手柄位于每行最左侧，鼠标样式为 `grab` / `grabbing`
- 拖拽中被拖行降低透明度（`opacity: 0.5`）
- 目标位置显示 2px 品牌色横线作为插入指示
- 拖拽结束后，行列表和日历甘特图同步刷新

### 数据变更

- 修改 `topic.productionSteps` 数组的元素顺序
- 调用 `reverseSchedule(publishDate)` 重新计算日期
- 保存到 `appState` 并触发 `saveData()`
- 记录操作日志 `step.reorder`

## R07 — 多段拍摄通告绑定

### 描述

当前 Canbox 通告绑定仅支持单段连续拍摄日期。当通告排期中存在多个不连续的拍摄时间段时（如 7.14-15 外景 + 7.22-23 棚拍），需要全部绑定并分别应用于排期。

### 当前行为

- 从 Canbox 拉取的项目按名称聚合，`dates` 数组包含所有拍摄日期
- 导入时取 `dates[0]` 和 `dates[dates.length-1]` 作为连续拍摄段
- 若实际拍摄日期不连续（如 7.14, 7.15, 7.22, 7.23），会被错误合并为 7.14~7.23

### 目标行为

1. **识别多段拍摄**：对 `dates` 数组进行连续性检测，将连续日期分组为独立拍摄段

   ```
   dates: [7.14, 7.15, 7.22, 7.23]
   → segments: [{start: 7.14, end: 7.15}, {start: 7.22, end: 7.23}]
   ```

2. **数据模型扩展**：`canboxImport.shootSegments` 数组

   ```json
   {
     "canboxImport": {
       "shootSegments": [
         { "start": "2026-07-14", "end": "2026-07-15", "label": "外景" },
         { "start": "2026-07-22", "end": "2026-07-23", "label": "棚拍" }
       ],
       "shootStart": "2026-07-14",
       "shootEnd": "2026-07-23"
     }
   }
   ```

   - `shootStart` / `shootEnd` 保留兼容性，为全部拍摄日期的最早/最晚
   - `shootSegments` 为新增字段，记录分段详情

3. **工序拆分**：多段拍摄时，拍摄工序拆分为多个独立色块

   - 方案 A（推荐）：`productionSteps` 中保留一个 `shoot` 工序，但其 `segments` 字段存储多段时间，甘特图渲染为多个色块
   - 方案 B：每个拍摄段生成独立工序 `shoot_1`、`shoot_2`，顺序排列

4. **双锚点倒排适配**：`anchorSchedule()` 函数需要支持多段拍摄

   - 拍摄前的工序：从第一段 `shootSegments[0].start` 往前倒推
   - 拍摄后的工序：从最后一段 `shootSegments[last].end` 往后顺推
   - 段与段之间的间隔：保持原样（不自动填充工序）

5. **UI 展示**

   - 甘特图中多个拍摄色块之间用虚线连接，表示同一工序的分段
   - 选题卡编辑页的拍摄工序行显示所有时间段，每段可独立编辑日期
   - 绑定通告信息区域显示所有拍摄段

6. **Canbox 导入流程**

   - `fetchCanboxProjects()` 中，对每个项目的 `dates` 数组进行连续性分组
   - 渲染项目列表时显示段数：`📅 7.14~15, 7.22~23 (2段)`
   - 导入/绑定时将 `shootSegments` 写入 `canboxImport`

### 向后兼容

- 已有的单段拍摄选题不受影响（`shootSegments` 为 `undefined` 时使用 `shootStart/shootEnd`）
- 渲染逻辑优先检查 `shootSegments`，不存在时 fallback 到现有逻辑
